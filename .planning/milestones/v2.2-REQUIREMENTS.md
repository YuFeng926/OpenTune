# Requirements: OpenTune v2.2 ARA2 线程模型与快照架构收敛

**Defined:** 2026-04-16
**Core Value:** 双格式独立编译，零交叉影响

本里程碑聚焦 VST3 分支的 ARA2 适配层收敛：按 ARA2/JUCE 官方契约，把当前分裂成多张 source 半状态表、多 getter 拼装真相、source-level mapping、立即读音频加 retry 的旧结构，重构为唯一正确的 `mutable model + immutable snapshot + serial copy worker + region-level playback truth`。当前范围只处理 `OpenTuneDocumentController`、`OpenTunePlaybackRenderer`、VST3 `PluginEditor` 及对应测试守护；不扩 shared processor 核心层，不保留兼容 getter、retry 或并行旧新结构。

## Current Milestone Requirements

### 快照状态模型

- [x] **SNAP-01**: 宿主完成一次 ARA model edit 批次后，VST3 ARA 读侧只能看到一份自洽的 published snapshot，不再混读多张 mutable 状态表
- [x] **SNAP-02**: 同一 `ARAAudioSource` 下存在多个 `ARAPlaybackRegion` 时，插件按 region 维度维护 playback/source mapping，不再被 source-level 单槽映射污染
- [x] **SNAP-03**: 音频线程与 editor 线程都只读 immutable snapshot，不触碰 controller 的 mutable model

### ARA 生命周期契约

- [x] **LIFE-01**: 插件不会再在 `didUpdateAudioSourceProperties()` 中立即读全量音频，也不会通过延时 retry 猜测 sample access
- [x] **LIFE-02**: 宿主启用或禁用 sample access 时，插件只通过 `will/didEnableAudioSourceSamplesAccess()` 驱动或终止音频 copy 生命周期
- [x] **LIFE-03**: `doUpdateAudioSourceContent()` 只标记 content dirty 与 revision 变化，真实内容发布延后到 `didEndEditing()` 后统一决策
- [x] **LIFE-04**: 宿主移除 playback region 或 audio source 后，published snapshot 立即去除对应 stale region/source/binding

### 读侧消费收敛

- [x] **CONS-01**: `OpenTunePlaybackRenderer` 每个 block 只读取一次 snapshot，并按 host 提供的 playback region 直接解析播放真相
- [x] **CONS-02**: VST3 editor timer 只按 snapshot epoch 判断是否需要同步，不再通过多 getter 拼装当前 ARA 真相
- [x] **CONS-03**: 用户手动导入当前 ARA region 时，editor 只从 preferred region view 读取音频副本与 mapping，不再依赖 `currentPlaybackAudioSource` 或 retry
- [x] **CONS-04**: 同一 source 下切换 preferred region 时，即使 content 未变且 `mappingRevision` 数字相同，editor 仍能触发正确 mapping sync

### 清理与回归守护

- [x] **CLEAN-01**: 旧 `AudioSourceState`、binding/sync/range getter、retry 和 source-level truth 相关生产代码全部移除，不保留兼容层
- [x] **CLEAN-02**: 测试与静态检查不再引用已删除结构；指定 grep 关键字结果全部为零
- [x] **CLEAN-03**: 本里程碑保持现有 `OpenTuneAudioProcessor` 导入/替换/局部重渲染/统一 playback read API 不扩 core，且不破坏 Standalone 分支行为

## Future Requirements

### 后续扩展

- **TEST-02**: 为多个 ARA 宿主建立更完整的自动化回归矩阵，不只覆盖当前 live tree 的最小结构守护
- **PERF-02**: 在结构收敛完成后，再基于测量结果评估 copy worker 调度与 snapshot 发布的性能优化空间

## Out of Scope

| Feature | Reason |
|---------|--------|
| 扩展与 ARA2 适配层收敛无关的 Standalone UI、新宿主功能或新产品能力 | 本 milestone 只处理 VST3 ARA 结构收敛 |
| 改动 `OpenTuneAudioProcessor` 核心抽象、增加第二套 playback/read/render 队列或新的调度系统 | live tree 现有 processor API 已够用 |
| 保留 `getAudioSourceState()` / `getPlaybackRangeForAudioSource()` / `getSourceRangeForAudioSource()` 等兼容 getter、`retry` 或 source-level playback truth | 这些就是本轮要删除的错误结构 |
| 把 sample access 生命周期继续建立在延时猜测、shared reader 或跨线程 reader 共享上 | 与 ARA2 官方契约冲突 |
| 借由“最小改动”保留并行旧新结构、兼容层、兜底路径或 message-thread 调度系统 | 本 milestone 目标是结构归一，不是止血 |
