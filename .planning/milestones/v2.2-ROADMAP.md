# Roadmap: OpenTune VST3 合并

## Overview

路线图按里程碑连续编号维护。最近完成的 milestone v2.1 聚焦 `PianoRollComponent` 刷新体系单环收敛：删除 overlay、多入口 repaint 和 heartbeat/timer 视觉路径，把钢琴卷帘统一到唯一正确的 `单入口 + 单层 + 单 VBlank 时钟` 结构。当前 milestone v2.2 聚焦 VST3 ARA2 适配层的线程模型与快照架构收敛：用 `mutable model + immutable snapshot + serial copy worker + region-level playback truth` 取代 source-level 单槽映射、多 getter 拼装真相和 retry 读音频残留，同时显式守住 Standalone 优先、双格式 UI 隔离和 shared processor 边界不扩 core。

### ✅ v2.2 ARA2 线程模型与快照架构收敛

**Milestone Goal:** 按 ARA2/JUCE 官方契约，把 VST3 分支的 ARA 适配层收敛为最小正确线程模型与状态发布结构。

**Overview:** 这个 milestone 不扩 shared `OpenTuneAudioProcessor` 核心抽象，也不保留兼容 getter、retry 或并行旧新结构。阶段顺序明确为：先把 `OpenTuneDocumentController` 内部状态压成 `sources_ + regions_ + preferredRegion_ + publishedSnapshot_`，让 region-level playback truth 取代 source-level 单槽 mapping；再把 sample access 生命周期收敛到 callback 驱动、把 snapshot publication 固化为 renderer/editor 唯一读约定；随后让 renderer 与 VST3 editor 只消费 snapshot / epoch；最后删除旧 getter、retry、binding/sync 半状态并用 tests + grep-zero 守住 Standalone 与 shared processor 双分支正确性。

- [x] **Phase 24: Callback 驱动的 sample access 生命周期** - 按官方 callback 重新分工 copy / dirty / publish / removal，不再立即读音频或 retry 猜测 access (completed 2026-04-16)

## Phase Details

### Phase 11: 最小状态模型定稿
**Goal**: 冻结 VST3/ARA clip 渲染链路的最小状态模型，让后续实现只围绕 `clipId / renderRevision / mappingRevision / layoutEpoch / RenderCache / RenderTask` 展开
**Depends on**: Phase 10 code baseline（v1.1 剩余人工验收不作为本 phase 阻塞）
**Requirements**: TRUTH-01, TRUTH-03, SAFE-03
**Success Criteria** (what must be TRUE):
  1. 规划与实现契约都以 `clipId` 作为 VST3/ARA clip 渲染链路唯一身份键，不再把 `track`、`audioSourceId` 或 `playbackRegionId` 写成语义主角。
  2. `mapping-only` 变化被明确定义为“保持同一 `clipId` 语义，只更新读取映射”，不再允许通过更换 clip 身份逃避版本语义。
  3. 文档与后续 phase 目标都只围绕六个最小抽象展开，不再引入并行旧新结构或第二套状态模型。
**Plans**: TBD（由 `/gsd-plan-phase 11` 生成）

### Phase 12: Clip 真相源与映射收敛
**Goal**: 让 ARA/VST3 播放链路先统一解析到 `clipId`，再通过 `clipId -> RenderCache` 读取；同时把 mapping-only 变化从 render queue 彻底隔离
**Depends on**: Phase 11
**Requirements**: TRUTH-02, REV-01
**Success Criteria** (what must be TRUE):
  1. ARA 播放读取事实源统一命中 `clipId -> RenderCache` 绑定，不再依赖 adapter 私有 cache、`AudioSourceState::renderCache` 旁路或 `track 0` 约定。
  2. `mapping-only` 变化只 bump `mappingRevision` 并更新映射，不触发 render queue invalidation，不污染旧 revision 去重语义。
  3. clip 真相源收敛完成后，VST3/ARA 播放路径不存在第二个可写读取事实源。
**Plans**: TBD（由 `/gsd-plan-phase 12` 生成）

### Phase 13: Revision / Epoch 驱动的旧队列收敛
**Goal**: 保持旧 `RenderCache / pending / revision` 队列核心不变，只让 `render-affecting` 进入它，并用 `layoutEpoch` 阻断 stale publish
**Depends on**: Phase 12
**Requirements**: REV-02, REV-03, QUEUE-01, QUEUE-02, QUEUE-03
**Success Criteria** (what must be TRUE):
  1. 旧 `RenderCache / pending / revision` 队列核心继续承担渲染调度，不引入第二套队列、事件总线、兼容层或双写路径。
  2. 只有 `render-affecting` 变化会 bump `renderRevision` 并进入旧 revision 队列；mapping-only 不入队，且只失效受影响 chunk。
  3. `layout-affecting` 变化先 bump `layoutEpoch` 并触发 ARA reread；只有 reread 形成新的渲染输入后，才派生一次 render-affecting invalidation。
  4. 旧 `RenderTask` 在新的 `layoutEpoch` 上不能 stale publish；发布必须校验 `clipId + renderRevision + layoutEpoch`。
**Plans**: TBD（由 `/gsd-plan-phase 13` 生成）

### Phase 15: 样本真相源与冻结边界
**Goal**: 让 shared core 从导入、chunk 任务到 worker/callback 全部只认 `44.1kHz + sample index` 冻结边界，不再让 seconds 在下游重新生成事实
**Depends on**: Phase 14 code baseline
**Requirements**: SAMP-01, SAMP-02, TASK-01, TASK-02, TASK-03
**Success Criteria** (what must be TRUE):
  1. 用户导入 clip 后，系统在 `44.1kHz` 内部工作域中保存真实 sample 长度与边界，后续播放、渲染和缓存流程继续消费同一组 sample 事实。
  2. 用户触发 chunk 渲染时，worker、callback 和 cache 流程读取的是同一组冻结字段（`trueStartSample / trueEndSample / synthEndSample / publishSampleCount / synthSampleCount / frameCount`），而不是各自再从秒值重算边界。
  3. 用户在 worker、callback 和 cache 路径之间，不会再遇到模块用 `seconds` 自行推导长度去替代冻结契约。
  4. 用户命中输出长度与 `synthSampleCount` 不一致的 chunk 时，任务会失败，且不会发布任何不一致的缓存结果。

### Phase 16: Hop 对齐切分与末块语义
**Goal**: 让静息处切分、内部 chunk 边界、真实 `clipEndSample` 与 inference-local padding 在 sample 域下形成单一正确语义
**Depends on**: Phase 15
**Requirements**: CHNK-01, CHNK-02, CHNK-03, CHNK-04, TASK-04
**Success Criteria** (what must be TRUE):
  1. 用户的静息处切分边界在内部以 sample 表达，秒值不再作为最终 chunk 事实边界。
  2. 用户命中中间 chunk 时，切分点优先落在可整除当前 vocoder `hopSize` 的 sample 位置；当前静息处范围内没有合法点时，系统会跳过该静息处并继续寻找后续候选点。
  3. 用户命中最后一个 chunk 时，真实 `clipEndSample` 保持不变；只有 mel/vocoder 输入准备阶段允许补零到完整 hop。
  4. 用户处理 padded last chunk 时，人工 padding 产生的尾部 frame 不会再被解释成 voiced F0 延伸。
